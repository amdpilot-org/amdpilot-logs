## Task Description

# [ROCM] Optimize fused_topk_bias to use aiter instead of fallback torch ops

## Bug Description

When running MiniMax-M2.5 (a Mixture-of-Experts model with 256 experts) on ROCm with `VLLM_ROCM_USE_AITER=1`, the `fused_topk_bias` function in vLLM falls back to generic PyTorch ops for the biased top-k routing computation. This fallback path is significantly slower than using optimized aiter kernels, causing a measurable throughput degradation (~14% slower TPOT).

The issue is that `fused_topk_bias` lacks a fast path to leverage aiter's `biased_grouped_topk` kernel when sigmoid scoring with bias correction is used. The aiter kernel has a hard limit of 32 experts per group, and models like MiniMax-M2.5 with 256 experts need the group count computed correctly to satisfy this constraint.

## Expected Fix

Add a fast path in `fused_topk_bias` that:

1. Detects when running on ROCm with aiter enabled and sigmoid scoring with bias correction
2. Computes the smallest `num_expert_group` that satisfies `num_experts / num_expert_group <= 32` and evenly divides `num_experts`
3. Sets `topk_group = num_expert_group` to make it semantically equivalent to a global top-k with bias correction
4. Routes to aiter's `biased_grouped_topk` kernel instead of the generic Python fallback

The key file to modify is:
- `vllm/model_executor/layers/fused_moe/router/fused_topk_bias_router.py`
