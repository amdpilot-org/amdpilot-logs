  Problem

  The wvSplitK skinny GEMM kernel in vLLM's ROCm backend assumed contiguous memo
  ry layout (stride == K) for both activation and weight tensors. Padded (non-co
  ntiguous) tensors were blocked by an is_contiguous() guard in the routing logi
  c, forcing them to use the slower torch.nn.functional.linear fallback.

  Solution

  Following the pattern already established by the FP8 wvSplitKQ kernel variant:

  1. Kernel signatures (csrc/rocm/skinny_gemms.cu): Added Kap (activation stride
     nd Kbp (weight stride) parameters to all three kernel variants:
    • wvSplitK_hf_sml_ — small A fits entirely in LDS
    • wvSplitK_hf_ — medium A, partially in LDS
    • wvSplitK_hf_big_ — large A with PCML chunked loading
  2. Memory access patterns: Updated all stride-dependent memory accesses:
    • A (activation) loading to LDS: Changed from flat A[k_in] to stride-aware A
  * Kap + col]
    • B (weight) loading: Changed from B[(m+y) * K + k_] to B[(m+y) * Kbp + k_]
    • A memory fallback reads: Changed from A[k_ + K * n] to A[k_ + Kap * n]
    • LDS-internal reads remained unchanged (LDS is packed densely using K)
  3. Host function: Extracted strides using in_a.stride(0) and in_b.stride(0) an
     assed them through the dispatch macros.
  4. Routing guard (utils.py): Removed x.is_contiguous() from the use_skinny con
     ion while keeping it for the use_skinny_reduce_counting path (which was not
     dified).
